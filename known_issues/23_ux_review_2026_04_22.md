## Global / Cross-cutting

### Navigation & layout

#### HIGH: Long Settings pages have no sub-navigation — entire screens must be scroll-hunted

**Location:** Settings → Purchasing, Settings → Accounts, Settings → Users & API Keys.

**Evidence:** The Purchasing and Users pages each span multiple viewport heights with four+ sub-sections. There is no sticky left rail, anchor bar, or accordion to jump between sections like `General Defaults → AWS → Azure → GCP → Exchange Automation`. Users scroll to find the knob they want.

**Recommended fix:** Introduce a sticky left sub-nav (or top anchor-link pill bar) on Settings pages with section headings linked. Backed by IDs already present on `<section>` elements, this is CSS + a few lines of JS in `navigation.ts`.

#### HIGH: Destructive "Delete" sits flush next to "Overrides" with no visual distinction — misclick risk

**Location:** Settings → Accounts, per-provider account rows.

**Evidence:** Account rows render five buttons inline: `Edit | Test | Credentials | Overrides | Delete`. All use the same outlined-button style. Delete is right of Overrides — a single column apart. Misclicks on a long row are easy.

**Recommended fix:** Either (a) style destructive buttons red (filled or outlined), (b) move Delete into an overflow `⋮` menu, or (c) require a confirmation modal. Prefer (a)+(c) together.

#### MEDIUM: No breadcrumbs — users on deep settings sub-pages have no orientation

**Evidence:** Settings → Users & API Keys → (expanded user row) has no breadcrumb, no "you are here" indicator beyond the two-level tab bar.

**Recommended fix:** Add a simple breadcrumb above the main content area for two-level pages: `Settings › Users & API Keys`. Native `nav[aria-label="breadcrumb"]` + CSS; no state changes needed.

#### MEDIUM: Content width is uncontrolled on wide monitors — tables stretch edge-to-edge with huge inter-column gaps

**Evidence:** Every page rendered at the screenshot capture width (~1400px+) has tables and forms spanning full width. On larger monitors (1920/2560px), inter-column whitespace becomes distracting and data density drops.

**Recommended fix:** Cap main content at ~1400–1600px centred; let the top nav bar remain full-width. Affects the top-level layout wrapper in `index.html` / `styles.css`.

#### MEDIUM: No global search

**Evidence:** Dashboards with many accounts / recommendations / plans typically offer `⌘K` or a header search for jumping to a specific entity. Absent here.

**Recommended fix:** Deferred — this is a full feature. File as a follow-up rather than inline fix. Worth flagging because its absence forces users to tab-hop for any cross-page lookup.

#### MEDIUM: Button hierarchy has no consistent semantic system

**Evidence:** Across the app the same visual weight is used for primary, secondary, and destructive actions (`Refresh`, `Load History`, `Reset to Defaults` — all dark grey filled) while the Dashboard uses `[Create Purchase Plan]` in blue for a primary. No clear convention: primary = blue filled, secondary = outlined, destructive = red / confirm-gated.

**Recommended fix:** Define four button variants in CSS — `.btn.primary`, `.btn.secondary`, `.btn.destructive`, `.btn.tertiary` — and sweep call sites. Load History should go primary-blue; Reset to Defaults should go destructive-outlined + confirmation.

#### MEDIUM: No "unsaved changes" indicator on Settings pages

**Evidence:** Long settings pages allow edits to many fields, but navigating to another tab loses them silently. No confirmation dialog, no dirty-state badge on the tab, no sticky save bar.

**Recommended fix:** Track dirty state per settings form; on tab change, show a "You have unsaved changes" dialog with Save / Discard / Cancel. Also add a sticky "Save Settings" bar at the bottom of long pages that appears only when dirty.

#### LOW: Card-in-card border nesting flattens hierarchy

**Evidence:** Settings → Accounts nests bordered cards three deep (page card → Federation Setup card → provider-specific cards inside). Border weight is identical at every level, so users can't tell at a glance what contains what.

**Recommended fix:** Reduce inner border weight (or drop borders on the inner-most level), and use spacing + typography to communicate nesting.

#### LOW: Fieldset-style blue section titles cut across their card borders

**Evidence:** Several section headings ("Users", "Groups & Permissions", "Federation Setup") render with a background colour that overlaps the card border — the classic `<fieldset><legend>` look. It reads as a rendering bug.

**Recommended fix:** Either use a standard card-header pattern (header row above a single-bordered body) or properly use `<fieldset>`/`<legend>` with its default browser styling. Don't mix.

#### LOW: `(admin)` badge placement looks like part of the email

**Evidence:** Header reads `cristi@leanercloud.com (admin)` — the badge's spacing makes it look like a domain-tail suffix rather than a separate pill.

**Recommended fix:** Convert `(admin)` to a proper styled badge (pill background, distinct colour, small font). If a multi-tenant workspace switcher is planned, put it on the left of the user identity instead.

#### LOW: No session/freshness indicator

**Evidence:** Once logged in, the app gives no indication of session age or "last data refresh" (other than the Dashboard's subtle "Data from 13h ago"). Commercial cost tools are often left open for hours.

**Recommended fix:** Small "Session refreshes in Xm" indicator near the user email; silent OIDC refresh on click.

### Visual system

#### MEDIUM: Date/time format is inconsistent across pages

**Evidence:**

- Purchase History date pickers show `21.01.2026` (DD.MM.YYYY European).
- Users table shows `3/25/2026` (M/D/YYYY US).
- Account Registrations timestamps show `Apr 17, 2026, 12:06 AM` (verbose localised).
- Last-login column mixes `18h ago` (relative) with `3/25/2026` (absolute).

**Recommended fix:** Pick one canonical format per context — `YYYY-MM-DD` or localised `Apr 17, 2026` for display — and respect the user's locale consistently. A small helper in `utils.ts:formatDate` / `formatDateTime` already exists; extend it and sweep the call sites.

#### LOW: Currency format is inconsistent (`$0` vs `$0.00` vs `$0.00/hr`)

**Evidence:** Dashboard KPI cards strip trailing zeros; Purchase History cards always show two decimals.

**Recommended fix:** Pick `$X,XXX.XX` with trailing zeros; refactor `utils.ts:formatCurrency` and call it everywhere.

#### LOW: Section labels mix all-caps and sentence case

**Evidence:** Purchase History labels `PERIOD SAVINGS`, `AVG HOURLY SAVINGS`, `PEAK SAVINGS` are all-caps; Dashboard labels `Potential Monthly Savings`, etc. are sentence case.

**Recommended fix:** Unify to sentence case everywhere.

### Accessibility

#### MEDIUM: Green-only success indicators (checkmarks in Permission Overview, coverage colours)

**Evidence:** The Permission Overview matrix uses a green checkmark for granted permissions and an em-dash for absent, relying on colour to convey state. Coverage pills use `.util-green/.util-yellow/.util-red` alone.

**Recommended fix:** Pair colour with shape or text — e.g., `✓ Granted` / `– Not granted` in the matrix; coverage percentages with a leading icon. Meets WCAG 1.4.1 (not rely on colour alone).

#### MEDIUM: Icon-only action buttons have no visible labels

**Evidence:** Users table Actions column shows two unlabelled pill toggles. RI list row buttons include `Test | Credentials | Overrides` with no aria-labels. Federation Setup download cards don't show a download icon or tooltip.

**Recommended fix:** Add visible labels (where space allows) or `aria-label` attributes. For icon-only controls, always pair with a title tooltip.

#### LOW: Form-label to input distance becomes huge on wide screens

**Evidence:** "Enabled Providers" label sits on the left; its three checkboxes sit flush-right. On a wide monitor there is ~800px of whitespace between them.

**Recommended fix:** Cap the form width at ~900px centred on the page (GitHub/Linear pattern), or left-align inputs within a fixed-width column next to their label.

## Dashboard

#### HIGH: `Data from 13h ago` indicator is buried in secondary typography and not colour-coded by staleness

**Evidence:** Text sits in muted grey next to the Refresh button, same weight as the `Provider:` filter label. 13-hour-old cost data is meaningfully stale for a service billed hourly.

**Recommended fix:** Promote to a pill badge inside / adjacent to the KPI cards. Colour by age: green <3h · amber 3–12h · red >12h. `freshness.ts` already exposes the field; render change only.

#### MEDIUM: Empty KPIs masquerade as data — add zero-state context with CTAs

**Evidence:** `Active Commitments: 0 · $0/mo committed`, `Current Coverage: 0%`, `YTD Savings: $0 (from commitment purchases)` all render as normal data cards next to `Potential Monthly Savings $351`. A brand-new user sees four cards, three blank, no guidance.

**Recommended fix:** Treat the zero-commitment state as an onboarding sequence: one "You haven't purchased any commitments yet — start with [See Recommendations →]" banner across the KPI row, and de-emphasise the zero cards until at least one commitment exists.

#### MEDIUM: Current Coverage card shows `0%` / `Target: 80%` as plain text — no visual progress indicator

**Evidence:** The card renders identical layout to the other three KPIs, losing the "how far from target?" signal that a progress bar or ring would give.

**Recommended fix:** Add a horizontal progress bar under the percentage using the existing `.util-green/.util-yellow/.util-red` classes.

#### MEDIUM: "Potential Savings by Service" chart has a `Current Savings` legend entry with no bars

**Evidence:** Legend shows yellow "Potential Savings" and green "Current Savings"; chart renders only yellow bars because Current Savings is 0 everywhere. The unmet green entry looks like a broken-data state.

**Recommended fix:** Either render a hairline green baseline for 0 (explicit zero), or omit the `Current Savings` legend item until a bar exists.

#### MEDIUM: No time-series view of savings on the Dashboard

**Evidence:** The page shows current-state KPIs only. A FinOps tool's most common need is "what did we save last quarter?" — not on this page. The Purchase History page has hourly data but no visual trend.

**Recommended fix:** Add a sparkline or small line chart under the KPI row showing savings-over-time (last 90 days). Uses the same data Purchase History aggregates.

#### LOW: Provider / Account filters are disconnected from the chart — relationship unclear

**Evidence:** Dashboard filter bar has `Provider: All Providers` and `Account: All Accounts` at the top; chart below shows per-service totals. Does changing the filter re-scope the chart? No visual cue.

**Recommended fix:** Add a subtle "Filters apply to all widgets below" hint, or visually group the filters inside the chart card so scope is obvious.

#### LOW: Compute bar dwarfs all others — needs a log-scale toggle or top-N filter

**Evidence:** The Compute bar is ~$270/mo; cache/ec2/rds are under $50. Small values are visually invisible.

**Recommended fix:** Log-scale toggle, or sort descending + render only top 5 with an "N others: $X" rollup.

#### LOW: Chart X-axis mixes cross-provider buckets (`compute`, `cache`) with AWS-specific (`ec2`, `rds`)

**Evidence:** `compute` is a cross-cloud abstraction overlapping with `ec2`. Users assume mutual exclusivity; here they aren't.

**Recommended fix:** Stack bars by provider within each service bucket, or split every category per provider (AWS-EC2 / AWS-RDS / Azure-VM / GCP-ComputeEngine / …).

#### LOW: `Upcoming Scheduled Purchases` empty state is text-only

**Evidence:** Bare text `No upcoming scheduled purchases`. No illustration, no CTA.

**Recommended fix:** Add a small illustration + "[Schedule a purchase from a plan →]" link that navigates to Purchase Plans.

## Recommendations

#### MEDIUM: 12+ column table has no column-visibility toggle and overflows narrow screens

**Evidence:** On 13" laptops the table horizontal-scrolls; the right-most `[Purchase]` button is below the fold on first render.

**Recommended fix:** Column-visibility dropdown + persist to `localStorage`. Default-hide `Count`, `Term`, `Payment`. Or freeze the Actions column to the right edge.

#### MEDIUM: Checkboxes imply multi-select but no bulk-action toolbar appears, and `Create Purchase Plan` has ambiguous scope

**Evidence:** Every row has a checkbox. The `[Create Purchase Plan]` button sits top-left regardless of selection; it opens the modal with an empty plan if nothing is ticked. No select-all, no "N selected" header, no bulk Purchase.

**Recommended fix:** Wire selection to the button — disable it when zero rows selected; relabel to `Create plan from N selected` when rows are ticked. Show a sticky toolbar `N selected — [Add to plan] [Bulk purchase] [Clear]` above the table.

#### MEDIUM: No sort indicators on column headers — sortability is invisible

**Evidence:** Headers show bare text (`Monthly Savings`, `Upfront Cost`, `Count`). No sort arrows, no cursor affordance.

**Recommended fix:** Add sort arrows; make at least the numeric columns (`Monthly Savings`, `Upfront Cost`, `Count`) sortable.

#### MEDIUM: Long Resource Type names truncate with no tooltip

**Evidence:** `Azure_Redis_Cache_Premium_P1_Cache_Azure_Internal_CACHE` truncates mid-string. No `title` attribute; hovering shows nothing.

**Recommended fix:** Add `title={rec.resource_type}`. Zero-cost fix.

#### MEDIUM: No "why this recommendation?" detail view

**Evidence:** Rows show the what (instance type, savings, term) but no usage history, confidence score, or reasoning. A user about to commit $494 upfront has no way to drill in.

**Recommended fix:** Row-click opens a side-drawer with usage graph, utilization thresholds, confidence bucket, and "based on N days of CloudWatch/Cost Explorer data" provenance.

#### MEDIUM: `[Purchase]` on every row is tedious for bulk workflows

**Evidence:** 12 recommendations = 12 Purchase buttons, each opens a modal, each must be confirmed individually.

**Recommended fix:** Rename primary per-row action to `Add to plan` and push the actual purchase execution to a bulk flow from the plan. Or keep Purchase but ensure the bulk toolbar (above) covers the common case.

#### LOW: Provider badges in ALL-CAPS (`AWS`, `AZURE`) read as shouty

**Evidence:** Badge labels render as `AWS` / `AZURE` in all caps.

**Recommended fix:** Title-case (`AWS` / `Azure` / `GCP`), or switch to provider icons with the label as an aria-label.

#### LOW: `$0` savings rows should be filtered out or flagged

**Evidence:** One row (`Standard_B1ls`) shows $0 Monthly Savings. A "recommendation" with zero savings isn't a recommendation.

**Recommended fix:** Default-filter rows with zero or negative savings; add a "Show uneconomical recommendations" toggle if they're still useful for debugging.

#### LOW: Term format `3 year` (table) vs `3 Years` (Settings) — inconsistent

**Evidence:** Recommendations table shows `3 year`, `1 year`. Settings → Purchasing Default Term dropdown shows `3 Years`.

**Recommended fix:** Unify to `1 year` / `3 years` everywhere (or `1 Year` / `3 Years`). Single helper in `utils.ts`.

#### LOW: Filter row is cramped — Account dropdown squeezes

**Evidence:** 5 filters + a button on one line; Account dropdown gets compressed on mid-width screens.

**Recommended fix:** Left-side filter panel, or a `More filters` overflow menu that collapses low-usage filters (Region, Min Savings) into a secondary row.

#### LOW: `Account` column repeats the same account many times — needs grouping

**Evidence:** `Test Azure Joshua (b9b8672f...)` repeats on many consecutive rows. No row-grouping or collapsible account sections.

**Recommended fix:** Optional grouped-by-account view with collapsible sections, or render the account name only on the first row of a run.

#### LOW: No dismiss / snooze action on recommendations

**Evidence:** Users can only Purchase or ignore. Ignored recs stay forever.

**Recommended fix:** Add `Dismiss` / `Snooze for 30 days` per row with a reason field (for analytics). Requires backend state.

#### LOW: `Count` column — unit unclear (instances vs recommendations?)

**Evidence:** Values `1`, `2`, `3`, `4` in the Count column with no header tooltip.

**Recommended fix:** Add `title` / info icon on the header: "Number of instances this recommendation covers."

#### LOW: `Payback Period: 1.4 months` — calculation opaque

**Evidence:** KPI on the summary row; no explanation of formula (upfront / monthly savings).

**Recommended fix:** Info-icon tooltip: "Upfront cost divided by monthly savings."

#### LOW: No pagination for scale

**Evidence:** 12 rows fits today. At 100+ rows the scrolling becomes unwieldy.

**Recommended fix:** Virtualised scroll or paginator (25/50/100 per page).

## RI Exchange

#### HIGH: `All convertible RIs are well-utilized` empty state is actively misleading when the RI list is also empty

**Evidence:** Active Convertible RIs section: *"No active convertible Reserved Instances found"*. Adjacent Reshape Recommendations section: *"No reshape recommendations. All convertible RIs are well-utilized."* — an affirmative utilization claim that cannot be true when there are zero RIs.

**Recommended fix:** Conditional empty state in `frontend/src/riexchange.ts:renderRecommendations` — consult `currentRIs.length` before choosing the copy.

#### MEDIUM: Refresh button scope is unclear on a page with three data sources

**Evidence:** One `[Refresh]` top-right; three sub-sections (RIs / Recs / History).

**Recommended fix:** Rename to `[Refresh all]` and ensure it re-fetches all three; or drop the global button and add per-section refresh icons.

#### MEDIUM: Empty sections have no guidance — cold-start experience is bleak

**Evidence:** Three empty panels stacked with no explainer. A new user has no idea what "Convertible RI Exchange" means or why they should care.

**Recommended fix:** Add a short explainer banner above the panels with a "Learn about convertible RI exchange →" link and a screenshot/illustration of what the page will look like populated.

#### LOW: Exchange History window hardcoded to 12 months

**Evidence:** Heading reads `Recent RI exchange operations (last 12 months)`. No control to change the window.

**Recommended fix:** Date-range picker or preset dropdown (`Last 3 / 6 / 12 / 24 months | All time`).

#### LOW: Empty-state messages are passive (`No X found`)

**Evidence:** `No active convertible Reserved Instances found` / `No exchange history found` — passive voice, no action verb.

**Recommended fix:** Active voice with a next-step: `You haven't made any RI exchanges yet. When you do, they'll appear here.`

## Settings → Purchasing

#### HIGH: Every sub-panel uses the same literal `Settings` header — impossible to scan which provider you're configuring

**Evidence:** Four panels each titled `Settings` in a coloured bar (yellow/blue/purple/teal). Provider is only implicit via the service list below (EC2 = AWS, VM Reservations = Azure, etc.).

**Recommended fix:** Rename panel headers to `AWS Service Defaults` / `Azure Service Defaults` / `GCP Service Defaults` / `Exchange Automation`. Keep colour bars as secondary reinforcement.

#### HIGH: Cascading default changes are silent magic — no confirmation, no diff preview

**Evidence:** The page note says *"Changing these defaults will automatically update all compatible service settings below"*. Editing one top-level default silently rewrites N per-service rows, with no preview or undo.

#### MEDIUM: Dense N×3 grid with no "Same as global" affordance

**Evidence:** Every service has Term / Payment / Coverage trio. Configuring N services individually is tedious when most users want "same as global" with one or two overrides.

**Recommended fix:** Per-service "Same as global" checkbox that hides the service's own controls when checked; an "Override" button reveals them.

#### MEDIUM: Sticky save bar missing on a very long page

**Evidence:** `[Save Settings]` / `[Reset to Defaults]` are only at the bottom of the longest page in the app. Users editing fields mid-page often forget to scroll down and save.

**Recommended fix:** Dirty-state-aware sticky bar docked to the bottom of the viewport when unsaved changes exist.

#### LOW: `Service Defaults` grid wraps awkwardly at narrow widths

**Evidence:** AWS service cards render in 3–4 columns wide, wrapping to multiple rows on narrow viewports.

**Recommended fix:** Convert to a vertical table layout (`Service | Term | Payment | Coverage` columns).

#### LOW: Azure/GCP cards have no `Payment Option` field — absence is unexplained

**Evidence:** AWS service cards have Term + Payment + Coverage; Azure/GCP show only Term + Coverage. The top-of-page note explains why, but the layout doesn't (users scanning the grid notice asymmetry first, notes later).

#### LOW: `OpenSearch Reserved Instances` card is taller than its siblings because its title wraps

**Evidence:** Longer heading forces two lines; adjacent cards stay at one. Grid heights misalign.

**Recommended fix:** Truncate with tooltip, or fix card min-height.

#### LOW: `Exchange Automation` sub-fields stay editable when the automation toggle is off

**Evidence:** Toggle off → fields below look editable. Confusing signal.

**Recommended fix:** Grey + disable when toggle is off.

#### LOW: `Max Payment Per Exchange: 0` / `Max Payment Daily: 0` — `0` could mean "no limit" or "invalid"

**Evidence:** Bare `0` with no helper text.

**Recommended fix:** Helper text: "Set to 0 to disable automatic purchases; any positive value is the hard cap in USD."

#### LOW: `Lookback Days: 30` has the unit in the label not the input

**Evidence:** Other fields inconsistently place the unit in the input suffix vs the label.

**Recommended fix:** Always suffix the input (`[ 30 ] days`).

#### LOW: No per-provider save — changing one AWS default requires saving the entire page

**Evidence:** Single bottom save button covers all provider panels.

**Recommended fix:** Per-panel save buttons in each provider sub-section header; top-level button becomes "Save all."

#### LOW: `Configured (IAM Role)` credential status only shown on AWS

**Evidence:** AWS panel shows "Configured (IAM Role)" badge; Azure/GCP equivalents absent.

**Recommended fix:** Show credential-state badge for every provider panel with a consistent label (`Configured (Service Principal)` / `Configured (Workload Identity Federation)` / etc.).

#### LOW: Panels for providers disabled in General tab still render fully expanded

**Evidence:** Unchecking Azure in General tab doesn't collapse the Azure Service Defaults panel here.

**Recommended fix:** Collapse the panel and show `Azure Service Defaults — disabled. [Enable Azure →]`.

## Settings → Accounts

#### MEDIUM: `Account Registrations` table and per-provider `Accounts` panels have unclear relationship

**Evidence:** Top table shows self-service registration requests (`pending`/`approved`). Lower per-provider panels list accounts. Unclear whether approved registrations automatically appear in the per-provider list, whether the two stores are independent, or which is canonical.

**Recommended fix:** One-line explanation below the Registrations heading: "Approved registrations appear in the provider lists below." If they're separate data stores, rename to "Pending self-service registrations" vs "Active accounts."

#### MEDIUM: Actions column in Account Registrations is messy — three buttons stacked in two rows

**Evidence:** `Approve / Reject` on the top row, `Delete` on the bottom. No consistent horizontal layout.

**Recommended fix:** Put all three into a horizontal `Approve | Reject | ⋮` with Delete in the kebab menu; or use a standard three-button row with consistent widths.

#### MEDIUM: Per-provider Accounts sections render as inline text rows — need a table

**Evidence:** `AWS personal (540659244915) Edit Test Credentials Overrides Delete` is a single text-flow line. Hard to scan.

**Recommended fix:** Convert to a table: `Name | ID | Status | Actions`.

#### MEDIUM: 5 buttons per account row is too many primary affordances

**Evidence:** Every account has `Edit | Test | Credentials | Overrides | Delete` inline.

**Recommended fix:** Row-click opens a detail view; secondary actions (Test, Overrides) move into an overflow menu. Primary inline action = Edit; destructive Delete isolated.

#### MEDIUM: Timestamps are verbose — `Apr 17, 2026, 12:06 AM` eats column space

**Evidence:** Registrations table shows full locale timestamps; scanning for recency is hard.

**Recommended fix:** Show relative time (`5 days ago`) with absolute on hover, or a short absolute (`Apr 17`).

#### LOW: Federation Setup download options don't change per `Target cloud`

**Evidence:** Three download cards (`Terraform bundle`, `CLI script`, `CloudFormation`) regardless of which Target cloud tab is selected. CloudFormation is AWS-only; Azure wants Bicep/ARM; GCP wants Deployment Manager.

**Recommended fix:** Re-render the third card based on the selected Target cloud: CloudFormation (AWS) / Bicep (Azure) / Deployment Manager (GCP).

#### LOW: Federation Setup download cards don't look clickable

**Evidence:** No download icon, no hover state, no visible affordance that they trigger a download.

**Recommended fix:** Download icon + pointer cursor on hover + card-level click handler.

#### LOW: `External ID` column in Registrations shows full GUIDs — needs copy-to-clipboard

**Evidence:** `2d43f62c-4427-4f82-8db8-4cbf0f7f5da` wraps awkwardly.

**Recommended fix:** Truncate with a copy icon next to the shortened display.

## Settings → Users & API Keys

#### HIGH: API Keys section looks unfinished

**Evidence:** Only a `[Create API Key]` button visible; no list of existing keys, no "No keys yet" empty state, no rotation controls, no last-used timestamp, no scope/permission display.

#### MEDIUM: `Group Membership` radio buttons contradict multi-group reality

**Evidence:** User card shows `Group Membership` with radio buttons for each group — implying exactly-one membership. But the group-member counts elsewhere suggest users can be in multiple groups simultaneously.

**Recommended fix:** Replace radios with checkboxes. Backend already supports multi-group; ~5-line frontend change in `users/userModals.ts`.

#### MEDIUM: Users table Actions column shows unlabelled pill toggles

**Evidence:** Two small pills with no text; users can't tell what they do (edit? disable? revoke?).

**Recommended fix:** Add `aria-label` + visible icon (pencil / ban / trash) + hover tooltip.

#### MEDIUM: Expanded user row inline-edit has no visible Save/Cancel controls

**Evidence:** Expanding a user row shows editable Group Membership radios + Effective Permissions — but no Save/Cancel buttons on the expanded panel. Unclear if edits persist on collapse or require a separate action.

**Recommended fix:** Add `[Save] [Cancel]` in the expanded row, or make the group selector live-save with a toast confirmation.

#### MEDIUM: MFA "Disabled" in plain text — should be colour-coded

**Evidence:** Admin user's MFA column shows `Disabled` as plain text. For an admin account, disabled MFA is a security issue that deserves an amber or red status badge.

**Recommended fix:** `.status-badge.warn` (amber) or `.status-badge.danger` (red) for disabled; green for enabled.

#### MEDIUM: Technical permission strings (`view_recommendations`, `approve_plans`) shown to admins

**Evidence:** Permission pills and the Permission Overview matrix use snake_case technical names.

**Recommended fix:** Human-readable primary label (`View recommendations`) with the technical string as `title` tooltip for debugging.

#### MEDIUM: Permission Overview uses em-dash vs checkmark with colour-only differentiation

**Evidence:** `✓` for granted, `—` for absent. Relies on colour + symbol; no "Granted" / "Not granted" text. Also the `admin` row shows `✓` only under Administrators — but admin conceptually implies all other permissions.

**Recommended fix:** (a) pair symbols with text labels; (b) show inherited permissions with a distinct style (e.g., dotted checkmark for "granted via inheritance"); (c) make cells click-through to the group editor.

#### LOW: `admin:*` wildcard needs tooltip

**Evidence:** `Effective Permissions: admin:*` — opaque to non-technical admins.

**Recommended fix:** Tooltip: "Full access to all resources."

#### LOW: Permission tags not semantically grouped

**Evidence:** Listed in raw order — Read, Write, Approve, Admin mixed together.

**Recommended fix:** Group under collapsible headers within each card, or colour chips by category.

#### LOW: `No members` text for one group vs member pills for another — inconsistent

**Evidence:** Administrators shows member email pills below the permissions; Plan Authors shows plain `No members` text.

**Recommended fix:** Unify to the same pattern — empty state styled like the populated state (e.g., greyed pill `— No members`).

#### LOW: Permission Overview matrix not editable

**Evidence:** Matrix is read-only; changes require navigating to the per-group card.

**Recommended fix:** Click a cell to toggle the permission in-place, with a save confirmation.

#### LOW: Create User / Create Group / Create API Key buttons lack separation from their card titles

**Evidence:** Each button sits flush against its section heading with no visual separation.

**Recommended fix:** Add margin or move to a card-header pattern (title left, action right).
